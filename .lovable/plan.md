# 修复计划

## 已定位的问题

1. **已选信息出现在最后一条输出**
   - 不是模型乱序，而是前端把选项卡固定渲染在脚本上方，同时又把已选内容直接拼进 `brief.prompt`。
   - 结果：已选内容没有在原消息位置形成“确认摘要”，而是被后续输出/最终内容再次带出来，看起来像跑到了最后。

2. **服化道图片明明出来了却显示失败，且不该出现音色**
   - 服化道/人物图现在用 `onPartial` 先写入临时预览图，若最终上传或落库失败，会留下图片预览，但状态被标成 `Failed`，造成“有图但失败”的错觉。
   - 音色控件现在对 `wardrobe` 的 `W*` 也生效了；而且角色音色读取是**全局用户级**，没有严格按当前 task/project 过滤。

3. **任务音频并没有接真实后端**
   - 现在“任务音频”只是把 `assets` 里 `kind === audio` 或后缀像音频的资源筛出来。
   - 代码库里目前**没有**“对白 / 旁白 / 背景音乐”自动落到当前 task 的真实写入链路，也没有在分镜视频生成后自动分离并挂回任务的逻辑。

4. **视频经常失败、重试没用、成功结果没正确回到 UI**
   - 后端里已经存在该 task 的真实成功视频和真实 mp4 地址，但前端/轮询链路没有把所有成功状态稳定识别回来。
   - `pollVideoTask` 只按很窄的状态值做成功/失败归类，容易漏掉上游已完成但字段形态不同的返回。
   - 整段生成的自动降级逻辑（真人参考图被审查后改走 text-to-video）只在批量 `runLife()` 里有，**单段重试 `runLifeSegment()` 没有同步这套逻辑**，所以你点重试基本还是原地失败。

## 实施步骤

### 1. 修正 brief 选项确认的时间线位置
- 不再把已选摘要只偷偷拼进 `brief.prompt`。
- 在用户提交选项卡的**原位置**插入一条明确的“已确认选择”消息。
- 顶部固定区只保留还在等待操作的卡片；已提交卡片不再重复漂浮显示。
- 下游脚本生成使用结构化偏好字段，不再依赖把偏好文本塞进最后输出里。

### 2. 修正服化道误失败与音色显示范围
- 把“临时预览图”和“最终成功图”分开处理，避免预览成功但最终落库失败时还显示成“生成成功的图”。
- 服化道阶段移除音色 UI 与自动绑定逻辑。
- 音色绑定只保留在**人物/场景素材阶段中的人物素材**，并且按当前 `taskId` / `projectId` 读取，避免串任务。

### 3. 把“任务音频”接成真实任务数据
- 让“任务音频”只读取**当前 task** 的真实音频资源，不再混全局或其它任务数据。
- 为音频资源建立明确类型：`dialogue` / `narration` / `bgm`。
- 在分镜视频完成后，把真实返回的音频结果写入当前 task；如果上游暂时只给单轨，也先真实落库并在 UI 正确展示，再补自动分离链路。
- 若现有表结构不够表达音轨类型/来源，就补最小必要的数据结构与访问规则。

### 4. 修正视频状态识别、真实 URL 回填与单段重试
- 扩展 `pollVideoTask` 的成功识别，兼容上游不同完成状态与结果字段，不再只盯住单一 `success + oss_url` 形态。
- 发现后端已经成功但本地 placeholder 仍失败/处理中时，主动把真实 mp4 URL 回填到当前资产。
- 给 `runLifeSegment()` 补齐与批量生成一致的：
  - 真人参考图审核失败自动降级
  - 正确错误码归类
  - 重试后重新关联真实任务与真实视频
- 去掉“看起来像视频但其实不是当前真实结果”的状态错配。

### 5. 验证
- 验证选项确认后，摘要留在正确位置，不再跑到最后。
- 验证服化道成功图不会再挂 `Failed`，且不再显示音色。
- 验证人物阶段音色只作用于当前 task 的人物素材。
- 验证任务音频能看到当前 task 的真实音频条目。
- 验证分镜视频成功时直接拿到真实可播放 mp4；失败时错误信息与重试结果一致。

## 技术说明

- 我已经确认：后端当前是正常的，问题主要在**前端状态机 + 轮询解析 + 任务作用域**，不是单纯 UI 样式问题。
- 我也确认：数据库里已经有这个 task 的真实成功视频记录，所以这次不是“后端完全没生成”，而是“生成成功后没有稳定映射回当前界面”。
- 任务音频这块目前确实不是样式没做，而是**后端链路本身缺失**，这部分会按真实数据链路补齐。
